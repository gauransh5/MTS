# Cloudflare Workers Deployment

## How it works

The site is deployed as a **Cloudflare Worker** using [OpenNext](https://opennext.js.org/cloudflare). OpenNext compiles the Next.js app into a Worker-compatible bundle (`.open-next/worker.js`) and deploys it via Wrangler.

Static assets (images, fonts, JS chunks) are served directly from Cloudflare's asset store ā€” not through the Worker ā€” so they're fast and don't count against CPU time.

---

## Key files (all committed, all required)

| File | Purpose |
|------|---------|
| `wrangler.jsonc` | Cloudflare Worker config: name, entry point, bindings, observability |
| `open-next.config.ts` | OpenNext adapter config; its presence tells Wrangler to skip the migration wizard |
| `next.config.ts` | Next.js config; includes `outputFileTracingRoot` (required by OpenNext) and a dev-only OpenNext init import |

### `wrangler.jsonc` bindings explained

- **`ASSETS`** ā€” serves static files from `.open-next/assets/` at the edge
- **`WORKER_SELF_REFERENCE`** ā€” lets the Worker call itself (used by OpenNext middleware routing); must match `"name"`
- **`IMAGES`** ā€” Cloudflare image optimization for `next/image`

**Critical:** `"name"` in `wrangler.jsonc` must exactly match the Worker name in the Cloudflare dashboard (`murphytechstudio`). If they differ, the `WORKER_SELF_REFERENCE` binding will reference a non-existent Worker and deploys will fail with error code 10143.

## Cloudflare build settings

Configured in Workers & Pages ā†’ murphytechstudio ā†’ Settings ā†’ Build:

| Setting | Value |
|---------|-------|
| Build command | `npm run build` |
| Deploy command | `npx opennextjs-cloudflare build && npx opennextjs-cloudflare deploy` |
| Version command | `npx opennextjs-cloudflare upload` |

**Do not use `npx wrangler deploy` as the deploy command.** That command runs a migration wizard which auto-generates `wrangler.jsonc` from `package.json`, overwriting the committed config and potentially using the wrong worker name.

## Local scripts (package.json)

```bash
npm run build    # next build only (for local checks)
npm run deploy   # opennextjs-cloudflare build + deploy to production
npm run preview  # opennextjs-cloudflare build + preview locally
npm run upload   # opennextjs-cloudflare build + upload (non-production version)
```

## Adding a new page

No deployment config changes needed. Add the page under `app/`, run `npm run build` locally to verify, then push to `main`. Cloudflare auto-deploys on push.

## Adding Cloudflare bindings (KV, D1, R2)

1. Create the resource in the Cloudflare dashboard
2. Add the binding to `wrangler.jsonc` (e.g. `"kv_namespaces": [...]`)
3. Add the type to `cloudflare-env.d.ts` (generated by `npm run cf-typegen`)
4. Commit both files

## Updating observability or config

Edit `wrangler.jsonc` directly and commit. The next deploy picks it up. Do not change config through the Cloudflare dashboard without syncing it to `wrangler.jsonc` ā€” the file is the source of truth.

## Domain

- Workers URL: `murphytechstudio.gauransh5.workers.dev` (always live)
- Custom domain: `murphytechstudio.com` (configured in Cloudflare DNS)

If `murphytechstudio.com` shows stale content after a deploy: go to Cloudflare Dashboard ā†’ Caching ā†’ Purge Everything.

## Troubleshooting

| Error | Cause | Fix |
|-------|-------|-----|
| `WORKER_SELF_REFERENCE ... not found [code 10143]` | `"service"` in `wrangler.jsonc` doesn't match `"name"` | Make both `murphytechstudio` |
| `Missing entry-point to Worker script` | `"main"` missing from `wrangler.jsonc` | Ensure `"main": ".open-next/worker.js"` is present |
| `npm ci` lock file mismatch | `package-lock.json` generated with `--legacy-peer-deps` | Run `rm -rf node_modules package-lock.json && npm install`, commit updated lock file |
| Build cache serving stale config | Old auto-generated files cached on build server | Clear build cache in Cloudflare: Settings ā†’ Build ā†’ Clear Cache |
